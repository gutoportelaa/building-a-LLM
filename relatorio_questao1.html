1 Objetivo e Formulação da Questão

A Questão 1 consiste em aplicar continued pre-training (pré-treino continuado, também denominado DAPT — Domain-Adaptive Pre-Training) a um modelo de linguagem de pequeno porte sobre o corpus do Diário Oficial dos Municípios do Piauí (DOM-PI), avaliando se o modelo adquire conhecimento de domínio mensurável após o treinamento.

Definição formal

Dado um modelo de linguagem M pré-treinado no corpus geral da internet, aplica-se treinamento adicional sobre um corpus de domínio D (publicações oficiais municipais piauienses), minimizando a loss de modelagem de linguagem causal:

L = − (1/T) Σ log P(xₜ | x₁, …, xₜ₋₁ ; θ)

O objetivo é reduzir a perplexidade do modelo sobre textos do domínio jurídico-administrativo piauiense, sem destruir o conhecimento geral adquirido no pré-treino original (evitar catastrophic forgetting).

Nota metodológica: A abordagem é full fine-tuning (todos os pesos atualizados), não LoRA/QLoRA. A escolha segue os estudos de Gururangan et al. (2020) que demonstram que DAPT com pesos completos supera adapters para ganhos de domínio com corpora grandes.

4 Metodologia de Pré-treino Continuado

4.1 Packed Causal Language Modeling

Para maximizar a eficiência de GPU e eliminar padding, adotou-se a técnica de packing: documentos são concatenados com o token EOS entre si e divididos em blocos fixos de block_size=1024 tokens. Blocos parciais ao final do corpus são descartados.

[doc₁ tokens] [EOS] [doc₂ tokens] [EOS] [doc₃ tok... | bloco 1024 | bloco 1024 | ...]

O objetivo é o next-token prediction padrão: dado o input x[0:1023], prever x[1:1024].

Bug corrigido (Job 381): A implementação original emitia um bloco parcial final (ex: 928 tokens) que causava RuntimeError: stack expects each tensor to be equal size no default_collate. O bloco parcial foi descartado intencionalmente, conforme recomendação de prática padrão em packing.

4.2 LR Schedule Manual — Apêndice D (Raschka 2024)

O schedule foi implementado manualmente seguindo o Apêndice D do livro de referência, com três fases:

# Fase 1: Warmup linear
initial_lr = 0.01 × peak_lr
lr_now = initial_lr + step × (peak_lr - initial_lr) / warmup_steps

# Fase 2: Cosine decay
progress = (step - warmup_steps) / (total_steps - warmup_steps)
progress = min(progress, 1.0)  # sem wrap-around
lr_now = min_lr + (peak_lr - min_lr) × 0.5 × (1 + cos(π × progress))

# min_lr = 0.1 × peak_lr

Parâmetros do schedule

ParâmetroValorReferência

O clamp min(progress, 1.0) foi adicionado para evitar que o schedule continue decaindo além de total_steps, comportamento que ocorria na implementação original do Raschka.

4.4 Seleção do Melhor Checkpoint

O checkpoint best é salvo a cada save_every=500 steps usando a perda média do intervalo (window_loss), não o loss do batch atual. Isso evita selecionar um checkpoint de um batch acidentalmente fácil.

Bug corrigido (Job 381): A implementação original comparava avg_loss do batch atual no checkpoint. No step 500, um batch de fácil (loss=3,08) foi selecionado como best, resultando em avaliação de um modelo quase não-treinado. A correção usa window_loss / window_count calculado ao longo dos 500 steps do intervalo.

O held-out é avaliado com janela deslizante de max_len=512 tokens com stride de 256 tokens (50%). Cada chunk é avaliado independentemente e a perda é acumulada sobre todos os tokens não-padded. Isso garante cobertura total dos documentos sem truncagem arbitrária.

7 Métricas de Avaliação

7.1 Definições

Cross-Entropy (CE)

Medida de surpresa média do modelo por token:

CE = −(1/N) Σ log P(xₜ | x<ₜ)

Calculada com reduction="none" sobre tokens não-padded (atenção mascarada). Menor é melhor.

Perplexidade (PPL)

PPL = exp(CE)

Interpreta-se como o número de opções igualmente prováveis que o modelo considera em cada posição. PPL=13,93 (baseline) significa que o modelo é equivalente a escolher uniformemente entre ~14 opções por token. Menor é melhor.

Token Accuracy

TokenAcc = #{argmax(logits) == label} / N_tokens

Fração de posições em que o modelo escolhe o token correto via greedy. Complementa CE pois é não-contínua. Maior é melhor.

CE Benchmark (NLL da resposta)

Para cada par (pergunta, resposta_ref) do benchmark, calcula-se a NLL apenas sobre os tokens da resposta, condicionada ao prompt:

CE_bm = −(1/|resp|) Σ log P(resp_t | prompt + resp<t)

Mede se o modelo atribui alta probabilidade às respostas corretas de domínio. Menor é melhor.

10 Diagnóstico e Análise

10.1 Por que o DAPT falhou nesta configuração?

A causa raiz não é o LR em si — é a qualidade do corpus. O campo texto do config curated inclui resíduos de OCR corrompido que tornam o sinal de treino quase indistinguível de ruído. Isso cria um regime patológico:

  • Loss inicial alta (~10): O modelo não consegue prever os tokens corrompidos, então os gradientes são enormes e destrutivos desde o primeiro step
  • LR muito baixo (1e-6): Os gradientes grandes são amortecidos, mas o modelo também não consegue mover os pesos para nada útil — estagna em loss ~7,5
  • LR moderado (2e-6): O modelo "aprende" o ruído OCR, corrompendo parcialmente as representações originais — CE held-out sobe de 2,63 para 3,74
  • LR alto (5e-6): O modelo sobreescreve quase tudo com o padrão do ruído OCR — colapso completo das capacidades linguísticas gerais

11 Próximos Passos

Prioridade Alta — Corpus mais limpo

  1. Treinar com corpus de Teresina (dom-pi-teresina-2025): texto mais limpo, mesmo LR de 1e-6 a 2e-6, esperando loss inicial ~3–4 em vez de ~10
  2. Limpeza adicional do corpus geral: filtrar documentos com razão de caracteres alfanuméricos < 0,7 e remoção de sequências de OCR claramente corrompidas
  3. Re-extração OCR com pdfplumber + post-processamento para reduzir artefatos comuns (cabeçalhos, rodapés, números de página repetidos)

Prioridade Média — Estratégia alternativa

  1. LoRA em vez de full-FT: atualiza apenas ~1% dos pesos, reduz drasticamente o risco de esquecimento com corpus ruidoso
  2. Warm-up mais longo: aumentar warmup_fraction de 5% para 15–20% para amortizar os gradientes destrutivos da fase inicial
  3. LR ainda menor com mais épocas: 5e-7 por 3 épocas, monitorando CE no held-out a cada 1000 steps (early stopping)
  4. Instruction tuning sobre os 33 Q&A pós-DAPT bem-sucedido
Hipótese para o próximo experimento: Com o corpus de Teresina (texto mais limpo), a loss inicial cairá de ~10 para ~3–4, os gradientes serão informativos, e um LR de 1e-6 a 2e-6 deverá produzir melhoria mensurável na CE do held-out (Δ CE esperado: −0,1 a −0,3). A razão de tokens de Teresina em relação ao corpus geral é menor, mas a qualidade do sinal compensa.
